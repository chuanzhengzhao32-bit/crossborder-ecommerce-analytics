# 跨境电商经营分析项目实施方案

## 1. 项目目标

构建一套可复现的跨境电商经营分析项目，证明候选人能够：

- 整合平台后台、广告、财务和库存等多来源数据。
- 建立统一的数据字典和经营指标口径。
- 按周期输出店铺与 SKU 级销售、毛利、费用、贡献利润和库存分析。
- 使用 SQL、Power Query 和 Power BI 建立可持续刷新的分析流程。
- 将分析结论转化为推广、运营、采购和仓储行动。

## 2. 目标读者

- 招聘经理：判断候选人能否胜任跨境电商经营分析。
- 业务负责人：理解项目能解决哪些经营问题。
- 数据分析师：复核数据结构、指标口径和代码。

## 3. 项目范围

### 纳入范围

- 3 个跨境站点或店铺。
- 20-30 个 SKU。
- 6 个月订单、广告和库存数据。
- 人民币统一核算。
- 店铺、SKU、渠道和月份四类分析维度。

### 暂不纳入

- 用户级生命周期分析。
- 复杂广告增量归因。
- 汇率套期保值和税务核算。
- 供应链预测模型。
- SPSS 与 RPA。

## 4. 数据源设计

| 数据表 | 粒度 | 主要字段 | 业务用途 |
|---|---|---|---|
| orders | 一行一订单明细 | 日期、店铺、SKU、数量、收入、退款、国家 | 销售和有效订单分析 |
| products | 一行一 SKU | 类目、采购成本、售价、生命周期 | 商品成本与分层 |
| ad_spend | 一行一日-广告组-SKU | 消耗、曝光、点击、归因订单、归因收入 | 推广效率分析 |
| fees | 一行一订单 | 平台费、支付费、物流费、税费 | 贡献利润核算 |
| inventory | 一行一日-SKU-仓库 | 可售库存、在途、缺货标记、库存成本 | 周转、缺货与滞销分析 |
| exchange_rates | 一行一日-币种 | 币种、人民币汇率 | 金额统一 |

## 5. 数据模型

采用星型模型：

- 事实表：订单、广告、费用、库存。
- 维度表：日期、店铺、SKU、国家/区域、渠道。
- 订单与费用通过订单号关联。
- 广告通过日期、店铺、SKU、渠道关联到公共维度。
- 库存通过日期、SKU、仓库关联到公共维度。

详细关系将在 `docs/data_model.md` 中记录。

## 6. 指标体系

### 规模指标

- 销售收入
- 有效收入
- 订单数
- 有效订单数
- 销量

### 利润指标

- 商品成本
- 毛利与毛利率
- 平台费用
- 物流费用
- 广告费用
- 贡献利润与贡献利润率

### 推广指标

- 广告消耗
- ROAS
- CPC
- 转化率
- 有效订单成本

### 库存指标

- 库存金额
- 库存周转天数
- 缺货率
- 滞销库存金额
- 补货覆盖天数

## 7. 分析问题

1. 哪些店铺和 SKU 贡献了增长？
2. 哪些增长没有转化为贡献利润？
3. 广告费、物流费或退款分别吞掉了多少利润？
4. 哪些广告组合应该加预算、观察或暂停？
5. 哪些 SKU 存在缺货风险？
6. 哪些 SKU 已形成滞销库存？
7. 下月预算和库存应如何分配？

## 8. 项目阶段

### 阶段 A：数据与口径

- 下载、校验并清洗 UCI Online Retail II 真实交易数据。
- 生成成本、费用、广告和库存模拟扩展数据。
- 建立字段字典与指标字典。
- 编写数据质量规则。

验收：主键唯一、日期完整、金额可对账、指标公式可解释。

### 阶段 B：SQL 与经营分析

- 建立 SQLite 数据库。
- 编写总览、利润、推广和库存 SQL。
- 输出可审计的结果表。

验收：SQL 可以重复执行，结果与 Python / Excel 汇总一致。

### 阶段 C：Power BI

- 建立星型模型。
- 编写 DAX 指标。
- 构建四页看板。

验收：筛选器联动正确，数字与 SQL 对账，页面能回答业务问题。

### 阶段 D：管理层报告

- 输出 Executive Summary。
- 展示关键发现和证据。
- 给出有负责人和复盘指标的行动建议。
- 记录假设、限制和下一步问题。

验收：招聘方只读 README 和报告即可理解项目价值。

### 阶段 E：GitHub 发布

- 清理敏感信息和大文件。
- 补充运行说明和截图。
- 检查链接、代码与命名。
- 初始化 Git，创建提交并上传远程仓库。

验收：新环境可按 README 复现核心结果。

## 9. 最终展示故事

项目讲解采用五步：

1. 业务方不能只看销售额，需要同时判断利润和库存质量。
2. 将六类业务数据整理为统一星型模型。
3. 用 SQL 与 Power BI 定位增长、亏损、投放和库存问题。
4. 给出预算调整、商品优化、补货和清仓建议。
5. 说明模拟数据边界及真实业务中需要补充的验证。

## 10. 完成标准

- README 中没有不可验证的真实公司业绩表述。
- 所有数据均可由脚本生成或提供明确来源。
- SQL、Python 和 Power BI 核心指标一致。
- 至少包含 3 项带证据的关键发现。
- 每项建议包含对象、动作和复盘指标。
- 仓库没有个人信息、临时文件或敏感数据。
